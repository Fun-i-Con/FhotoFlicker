# 現状理解

## 目的

このドキュメントは、Photo Flick の現在の実装を理解するための整理資料である。

基本設計や詳細設計に入る前に、まず次の点を明確にする。

- このアプリが何をするものか
- どの画面があるか
- どのデータを扱うか
- どのような処理の流れになっているか
- 今後クリーンアーキテクチャ化するとき、どこが分離対象になりそうか

## アプリの概要

Photo Flick は、スマホで撮影した写真を「場所」と「工程」に紐づけて、フリック操作で簡単に仕分けして、保存するWebアプリである。

主な特徴は次の通りである。

- スマホで写真を撮影または選択する
- 撮影時に現在地から登録済みの場所を判定する
- 撮影した写真を一時的に未分類写真として保存する
- 写真を上下左右にフリックして工程に分類する
- 保存済み写真を場所・工程で絞り込んで確認する
- データはサーバではなくブラウザの IndexedDB に保存する
- Nuxt 3 の SPA として動作し、GitHub Pages 配信を想定している

## 技術構成

現在の主な技術構成は次の通りである。

| 項目 | 内容 |
| --- | --- |
| フレームワーク | Nuxt 3 |
| 言語 | TypeScript / Vue |
| 実行形態 | SPA |
| 保存先 | IndexedDB |
| 配信 | GitHub Pages |
| 位置情報 | Geolocation API |
| 撮影 | `input type="file"` + `capture="environment"` |

## 画面一覧

### ホーム

ファイル: `pages/index.vue`

役割:

- アプリの入口
- 場所登録、撮影、工程設定、写真一覧への導線を表示する
- 現在の工程設定を表示する
- 保存済み写真数と工程数を表示する

### 場所管理

ファイル: `pages/places.vue`

役割:

- 撮影場所を登録する
- 現在地を取得して緯度・経度を入力する
- 場所名、緯度、経度、判定半径を保存する
- 登録済み場所を編集・削除する

### 撮影

ファイル: `pages/capture.vue`

役割:

- 撮影前に場所を選択する
- 現在地から最寄りの登録場所を自動選択する
- 写真を撮影または選択する
- 選択した写真を未分類写真として保存する
- 保存後に分類画面へ遷移する

### フリック分類

ファイル: `pages/classify.vue`

役割:

- 未分類写真を表示する
- 上下左右のフリック操作を受け取る
- フリック方向に対応する工程へ写真を分類する
- 分類後に写真詳細へ遷移する
- フリックできない場合に備えて方向ボタンでも分類できる

### 工程設定

ファイル: `pages/categories.vue`

役割:

- 工程カテゴリを登録・編集・削除する
- 工程名、色、フリック方向を設定する
- 同じ方向に複数の工程が割り当てられないようにする

### 写真一覧

ファイル: `pages/photos/index.vue`

役割:

- 保存済み写真を一覧表示する
- 場所で絞り込む
- 工程で絞り込む
- 写真詳細へ遷移する

### 写真詳細

ファイル: `pages/photos/[id].vue`

役割:

- 保存済み写真を大きく表示する
- 場所、工程、方向、保存日時を表示する
- 写真を削除する
- 同じ工程の写真一覧へ移動する

## データ一覧

現在の主要な型は `types/photo.ts` に定義されている。

### Category

工程カテゴリを表す。

| 項目 | 内容 |
| --- | --- |
| `id` | カテゴリID |
| `name` | 工程名 |
| `color` | 表示色 |
| `direction` | 対応するフリック方向 |
| `createdAt` | 作成日時 |
| `updatedAt` | 更新日時 |

制約:

- `direction` は `up`, `right`, `down`, `left` のいずれか
- 同じ方向には1つの工程だけ設定できる

### Place

撮影場所を表す。

| 項目 | 内容 |
| --- | --- |
| `id` | 場所ID |
| `name` | 場所名 |
| `latitude` | 緯度 |
| `longitude` | 経度 |
| `radiusMeters` | 場所判定に使う半径 |
| `createdAt` | 作成日時 |
| `updatedAt` | 更新日時 |

### DraftPhoto

撮影後、まだ分類されていない写真を表す。

| 項目 | 内容 |
| --- | --- |
| `id` | 未分類写真ID |
| `image` | 写真Blob |
| `imageType` | 画像MIME type |
| `placeId` | 紐づく場所ID |
| `createdAt` | 作成日時 |

### PhotoItem

分類済み写真を表す。

| 項目 | 内容 |
| --- | --- |
| `id` | 写真ID |
| `image` | 写真Blob |
| `imageType` | 画像MIME type |
| `categoryId` | 工程カテゴリID |
| `direction` | 分類時のフリック方向 |
| `placeId` | 紐づく場所ID |
| `createdAt` | 保存日時 |

## IndexedDB構成

IndexedDBの処理は主に `composables/usePhotoDb.ts` に集約されている。

現在のDB名:

- `photo-flick-sorter`

現在のDBバージョン:

- `3`

object store:

| store | 内容 |
| --- | --- |
| `categories` | 工程カテゴリ |
| `places` | 場所 |
| `drafts` | 未分類写真 |
| `photos` | 分類済み写真 |
| `settings` | 初期化済みフラグなど |

主なindex:

| store | index |
| --- | --- |
| `categories` | `direction` |
| `places` | `createdAt` |
| `photos` | `categoryId`, `createdAt`, `placeId` |

## 主な処理フロー

### 1. 初回起動

1. 画面表示時に `db.init()` を呼ぶ
2. IndexedDBを開く
3. 必要なobject storeとindexを作成する
4. 工程カテゴリの初期データを投入する
5. 画面に必要なデータを読み込む

### 2. 場所登録

1. 場所管理画面を開く
2. 現在地取得ボタンを押す
3. ブラウザのGeolocation APIで緯度・経度を取得する
4. 場所名と判定半径を入力する
5. `places` storeに保存する

### 3. 撮影

1. 撮影画面を開く
2. 登録済み場所一覧を読み込む
3. 現在地を取得する
4. 現在地が登録場所の判定半径内にあれば、その場所を選択する
5. 写真を撮影または選択する
6. 写真Blobと場所IDを `drafts` storeに保存する
7. 分類画面へ移動する

### 4. フリック分類

1. 分類画面を開く
2. 最新の未分類写真を読み込む
3. 写真を画面に表示する
4. ユーザーが写真を上下左右にフリックする
5. 移動量から方向を判定する
6. 方向に対応する工程カテゴリを探す
7. `photos` storeに分類済み写真として保存する
8. `drafts` storeから未分類写真を削除する
9. 写真詳細へ移動する

### 5. 写真一覧

1. 写真一覧画面を開く
2. 場所一覧、工程一覧、写真一覧を読み込む
3. 必要に応じて場所・工程で絞り込む
4. Object URLを作成して画像を表示する
5. 画面を離れるときにObject URLを破棄する

### 6. 写真削除

1. 写真詳細画面を開く
2. 削除ボタンを押す
3. 確認ダイアログを表示する
4. `photos` storeから該当写真を削除する
5. 写真一覧へ移動する

## 現在の責務の集まり方

現在は `usePhotoDb.ts` に多くの責務が集まっている。

含まれている責務:

- IndexedDB接続
- DBマイグレーション
- 初期カテゴリ投入
- カテゴリCRUD
- 場所CRUD
- 位置情報から最寄り場所を探す処理
- 未分類写真の保存
- 写真分類
- 分類済み写真の取得・削除
- Object URL生成

今後クリーンアーキテクチャ化する場合、このファイルを一気に分割するのではなく、次の順で分けると理解しやすい。

1. UIやIndexedDBに依存しない純粋なロジックを分ける
2. ユースケース単位で処理を名前付けする
3. IndexedDB操作をRepository実装として分ける
4. 画面側はUseCaseを呼ぶだけに近づける

## 分離しやすい候補

### 場所判定ロジック

現在:

- `findNearestPlace()` の中に距離計算と最寄り場所判定がある

分離候補:

- 緯度経度から距離を計算する関数
- 現在地から判定半径内の最寄り場所を選ぶ関数

理由:

- UIにもIndexedDBにも依存しない
- テストしやすい
- クリーンアーキテクチャの練習に向いている

### フリック方向判定

現在:

- `pages/classify.vue` の `resolveDirection()` にある

分離候補:

- 始点と終点から方向を判定する関数
- しきい値を定数化する

理由:

- UIから切り出しやすい
- テストしやすい
- 分類ユースケースとは別の入力判定として整理できる

### 写真分類ユースケース

現在:

- `classifyDraft()` が `usePhotoDb.ts` にある

分離候補:

- 未分類写真を方向で分類するUseCase
- 方向に対応するカテゴリを取得する処理
- 分類済み写真を保存し、未分類写真を削除する処理

理由:

- アプリの中心的な業務処理
- 後でRepository Interfaceを導入しやすい

## 現時点の設計上の確認ポイント

基本設計へ進む前に、次の点を確認する必要がある。

1. このアプリは今後もサーバなしのローカル保存アプリでよいか
2. 写真データは端末内だけに保存されればよいか
3. 複数端末での同期は不要か
4. 写真のバックアップやエクスポートはMVPに含めるか
5. 工程は上下左右の最大4つでよいか
6. 場所未選択の写真保存を許可するか
7. 削除済みの場所・工程に紐づく写真をどう扱うか
8. 対象ブラウザはスマホのSafari/Chromeを主対象にするか

## 優先して決めること

すべてを最初に決めようとすると進みにくいため、優先度を分ける。

### 先に決める

- サーバなしのローカル保存アプリとして進めるか
- MVPに含める機能範囲
- 写真を場所未選択でも保存できるか
- 工程は上下左右の4つを基本にするか
- 対象ユーザーと利用シーン

### 後で決めてもよい

- 複数端末同期
- 写真のエクスポート
- 写真の圧縮・リサイズ
- PWA対応
- 認証
- サーバ保存

### クリーンアーキテクチャ学習として先に見る

- 場所判定ロジック
- フリック方向判定
- 写真分類処理

これらはアプリの中心処理であり、UIやIndexedDBから分ける練習に向いている。
